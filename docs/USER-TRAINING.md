## 模块一：系统介绍与登录

### 1.1 DMH 系统简介

DMH（Digital Marketing Hub）是一个数字营销中台系统，为企业提供完整的营销活动管理解决方案。高级功能模块包含：

* **海报生成**：一键生成活动推广海报，无需设计技能
* **支付配置**：灵活的支付方式（订金/全款），支持微信支付
* **订单核销**：扫码核销，快速确认用户到场

## 模块二：海报生成功能

### 2.1 功能介绍与应用场景

#### 什么是海报生成？

海报生成功能允许品牌管理员和分销商为活动快速生成推广海报，无需依赖设计师，降低营销成本，提高效率。

#### 应用场景

| 场景 | 说明 | 使用角色 |
|------|------|---------|
| 新活动发布 | 为新发布的活动立即生成宣传海报 | 品牌管理员 |
| 分销推广 | 分销商为推广的活动生成个性化海报 | 分销商 |
| 批量营销 | 为多个活动批量生成海报进行多渠道推广 | 品牌管理员 |
| 活动预热 | 在活动开始前生成海报进行预热宣传 | 品牌管理员 |

### 2.2 海报模板说明

系统提供 3 种预设海报模板：

#### 经典模板

* **特点**：简洁大方，信息清晰
* **适用场景**：商务活动、企业宣传、正式场合
* **设计风格**：白色背景，标准排版，突出活动信息

#### 简约模板

* **特点**：极简风格，突出主题
* **适用场景**：短期促销、快闪活动、潮流活动
* **设计风格**：纯色背景，少文字，强视觉冲击

#### 时尚模板

* **特点**：渐变背景，年轻活力
* **适用场景**：青年群体、潮流活动、社交媒体推广
* **设计风格**：渐变色背景，现代化排版，适合分享

### 2.5 常见问题解答

**Q1：海报生成失败怎么办？**

原因：

* 网络连接不稳定
* 后端服务异常
* 活动信息不完整

解决方案：

1. 检查网络连接
2. 确认后端服务正常运行（访问 http://localhost:8889/health）
3. 检查活动是否已配置完整信息
4. 联系技术支持

**Q2：海报生成频率有限制吗？**

是的，为防止滥用：

* 每个用户每分钟最多生成 5 张海报
* 超出限制会提示「请求过于频繁」
* 等待 1 分钟后即可重试

**Q3：可以自定义海报模板吗？**

当前版本提供 3 种预设模板。如需自定义模板：

* 联系技术团队进行定制开发
* 后续版本将支持用户上传自定义模板

**Q4：海报的尺寸是多少？**

* 尺寸：750 × 1334 像素
* 比例：9:16（适合微信分享）
* 格式：PNG
* 文件大小：约 500KB - 1MB

**Q5：生成的海报有有效期吗？**

* 海报永久有效，可重复使用
* 建议在活动期间内使用
* 二维码内容包含活动链接，长期有效

## 模块三：支付配置功能

### 3.1 支付模式说明

DMH 系统支持两种支付模式：

#### 订金模式

**适用场景**：

* 高价值活动（如培训课程、高端活动）
* 需要确认名额的活动
* 现场需支付余款的活动

**流程**：

```
用户支付订金 → 预订名额 → 现场支付余款 → 核销确认
```

**优势**：

* 降低用户决策门槛
* 确保用户到场率
* 现场收款灵活

#### 全款模式

**适用场景**：

* 标准化活动（如展览、演出）
* 无需现场确认的活动
* 价格固定的活动

**流程**：

```
用户支付全款 → 确认参与资格 → 直接核销
```

**优势**：

* 收款一次性完成
* 管理流程简单
* 用户无需现场支付

### 3.2 订金 vs 全款模式对比

| 对比项 | 订金模式 | 全款模式 |
|--------|---------|---------|
| **支付流程** | 两步（订金+余款） | 一步（全款） |
| **决策门槛** | 低（只需支付订金） | 高（需支付全款） |
| **到场率** | 高（已付订金） | 中等 |
| **管理复杂度** | 高（需现场收款） | 低 |
| **适用活动** | 高价值活动 | 标准化活动 |
| **收款时间** | 分两次 | 一次完成 |

### 3.7 注意事项

1. **支付金额验证**
   * 订金金额必须小于全款金额
   * 系统会自动验证金额合理性

2. **商户号配置**
   * 确保微信商户号正确
   * 商户号需要在微信支付平台申请

3. **回调地址**
   * 确保回调地址可被微信服务器访问
   * 建议使用 HTTPS 协议

4. **二维码刷新**
   * 支付二维码缓存 2 小时
   * 过期后自动刷新
   * 可手动刷新获取新二维码

5. **支付安全**
   * 所有支付请求都经过签名验证
   * 防止金额篡改和伪造请求

## 模块四：订单核销功能

### 4.1 核销码生成机制

#### 核销码格式

```
{订单ID}_{手机号}_{时间戳}_{签名}
```

**示例**：

```
123_13800138000_1643775200_abc123def456
```

**字段说明**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 订单ID | 唯一标识订单 | 123 |
| 手机号 | 用户下单时填写的手机号 | 13800138000 |
| 时间戳 | 二维码生成时间（Unix时间戳） | 1643775200 |
| 签名 | HMAC-SHA1 签名，防伪造 | abc123def456 |

#### 核销码生成时机

* 用户支付成功后自动生成
* 核销码永久有效
* 用户可在「我的订单」中查看

### 4.2 扫码核销操作

#### 用户端：获取核销码

**步骤 1：用户下单并支付**

* 用户在 H5 页面填写报名表单
* 支付活动费用

**步骤 2：查看核销码**

* 登录 H5 前端：`http://localhost:3100`
* 进入「我的订单」
* 找到已支付的订单
* 点击查看核销码

**步骤 3：保存核销码**

* 截屏保存核销码
* 或直接出示二维码

#### 工作人员端：扫码核销

**步骤 1：打开核销页面**

* 访问核销页面：`http://localhost:3100/verify`
* 或在品牌管理后台进入「核销页面」

**步骤 2：扫描二维码**

* 使用手机扫描用户出示的二维码
* 或手动输入核销码

**步骤 3：系统验证**

系统执行以下验证：

```
1. 订单是否存在
2. 订单是否已支付
3. 核销码签名是否正确
4. 订单是否已核销（防重复）
```

**验证结果**：

| 验证项 | 通过 | 失败 |
|--------|------|------|
| 订单存在 | ✅ 继续验证 | ❌ 提示订单不存在 |
| 已支付 | ✅ 继续验证 | ❌ 提示订单未支付 |
| 签名正确 | ✅ 继续验证 | ❌ 提示核销码无效 |
| 未核销 | ✅ 允许核销 | ❌ 提示订单已核销 |

### 4.6 权限说明

| 操作 | 允许角色 | 说明 |
|------|---------|------|
| 核销订单 | 品牌管理员、平台管理员 | 必须有品牌管理员权限 |
| 取消核销 | 品牌管理员、平台管理员 | 必须有品牌管理员权限 |
| 查询核销记录 | 品牌管理员、平台管理员 | 只能查看本品牌记录 |

### 4.7 注意事项

1. **核销码防伪**
   * 签名验证确保核销码无法伪造
   * 任何篡改都会导致验证失败

2. **重复核销**
   * 已核销的订单无法再次核销
   * 系统会提示「订单已核销」

3. **未支付订单**
   * 未支付订单无法核销
   * 系统会提示「订单未支付」

4. **核销记录**
   * 所有核销操作都会记录日志
   * 包括操作人、操作时间、操作内容

5. **性能要求**
   * 核销接口响应时间 < 500ms
   * 支持高并发场景
